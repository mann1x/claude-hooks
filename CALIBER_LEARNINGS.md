# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** `stop_guard` has three escape layers, checked in this order: (1) user-wrap-up markers on the last USER message (`skip_on_user_wrap_up`, default on — e.g. "compact the context", "wrap up", "save state", "/wrapup") short-circuit before any pattern scan; (2) meta-marker phrases in the assistant text; (3) quoted-span stripping. When wiring the guard into a new handler, thread the last user-role text through `check_message(..., last_user_message=...)`; passing only the assistant text disables the strongest escape and causes false positives on legitimate wrap-up requests.
- **[convention]** Do not commit unverified test files. When the user interrupts mid-authoring (a `test_*.py` written but never run green), `rm` it before staging the rest of the phase — running `pytest tests/ -q` and getting `<N> passed` is the bar for inclusion. Commit messages that claim a phase is done must reflect only files whose tests are currently green.
- **[pattern]** `tests/mocks/ollama.py::mock_ollama_generate(target="claude_hooks.<module>")` patches `urllib.request.urlopen` *inside the named module*, not globally. The `target` argument must match the module whose `urlopen` call you want to intercept — use `claude_hooks.reflect` when testing `reflect.py`, `claude_hooks.consolidate` for `consolidate.py`, `claude_hooks.hyde` for HyDE. Using the wrong target silently passes the real urlopen through and lets tests hit the network.
- **[pattern]** Coverage gate runs via conda: `conda run -n claude-hooks coverage run -m pytest tests/` then `coverage report`. Config lives in `pyproject.toml` under `[tool.coverage.run]` and `[tool.coverage.report]` — branch coverage enabled, `pgvector.py`/`sqlite_vec.py`/`detect.py` omitted (integration / interactive code). Target ≥ 80 % branch. Commit `requirements-dev.txt` + `pyproject.toml` together when bumping coverage deps.
- **[gotcha]** When mocking `McpClient` in provider tests, patch `claude_hooks.mcp_client.McpClient` — NOT `claude_hooks.providers.base.McpClient`. Providers import the client lazily inside their `_client()` method, so the name does not exist in `providers.base`'s namespace and the patch raises `AttributeError: module ... does not have the attribute 'McpClient'`. Same rule applies to any symbol imported inside a method vs at module top level.
- **[gotcha]** `patch("builtins.__import__", side_effect=ImportError(...))` is far too broad — Python's `logging` module imports helpers internally on every `log.error()` call, so error-path tests that trigger a log will crash with a cascading ImportError. Use a selective side_effect that only raises for the specific module prefix under test (e.g. `if name.startswith("claude_hooks.hooks."): raise ImportError("boom"); return real_import(name, *a, **kw)`) and pass everything else through the real importer.
- **[gotcha]** `patch("sys.stdout")` with the default `MagicMock` breaks Python's logging: the log handler compares `record.levelno >= hdlr.level` where `hdlr.level` becomes a MagicMock, raising `TypeError: '>=' not supported between instances of 'int' and 'MagicMock'`. Replace stdout with a real minimal class that defines `write(self, s)` and `flush(self)` (and raises from `write` if you want to simulate failure) instead of a MagicMock.
- **[pattern]** Tests that mutate the `claude_hooks` logger (adding/removing handlers, changing level) need a save/restore fixture — otherwise state leaks and later tests see unexpected handlers or log output. Snapshot `root.handlers[:]` and `root.level` in setup, restore in teardown, and `.close()` any handlers the test added. Avoid patching `logging.handlers.RotatingFileHandler` without this fixture; the module's global state will persist across tests.
- **[correction]** Don't declare a coverage plan "complete" just because the configured minimum (≥ 80 %) was met — check the `coverage report --skip-covered` gap list and surface it. User preference: push coverage further when achievable even at diminishing returns ("I want a reasonably complete test coverage"). In this project 92 % was reachable in one focused phase targeting the lowest modules; stopping at 81 % when 92 % is a few hundred lines of boundary-mocking away under-delivers.
- **[convention]** When adding `coverage.py` to a project, also add `.coverage`, `.coverage.*`, and `htmlcov/` to `.gitignore` — coverage writes these to the repo root by default and they'll otherwise be listed as untracked on every run.
- **[gotcha]** `claude -p /cost` is non-functional for Claude subscription users — it only prints `"You are currently using your subscription to power your Claude Code usage"` and exits with no token data. `/cost` itself is session-scoped and API-user oriented. For weekly/historical usage, walk the transcript JSONL files under `~/.claude/projects/*/<session>.jsonl`: each `{"type":"assistant"}` entry carries `message.usage` with `input_tokens`, `output_tokens`, `cache_creation_input_tokens`, and `cache_read_input_tokens`. `~/.claude/stats-cache.json` has only `messageCount`/`sessionCount`/`toolCallCount` — no tokens.
- **[gotcha]** `ccusage` (`npx -y ccusage@latest`) groups usage by calendar day in whatever tz you pass via `-z` and cannot honour a non-midnight reset boundary (e.g. Claude's Friday 10:00 CEST weekly reset). Use it for USD pricing and per-model breakdowns (`daily --breakdown`), but roll your own transcript walker when you need the exact weekly-limit window.
- **[env]** On pandorum, the `claude-hooks` checkout is at `C:\Users\manni\claude-hooks` — NOT `C:\Users\manni\dev\claude-hooks`. Sync with `ssh pandorum "powershell -NoProfile -Command \"cd C:\Users\manni\claude-hooks; git pull --ff-only origin main\""`. Guessing `dev\claude-hooks` fails with `"The system cannot find the path specified."`
- **[gotcha]** `claude -p /cost` is non-functional for Claude subscription users — it only prints `"You are currently using your subscription to power your Claude Code usage"` and exits with no token data. `/status` and `/stats` are not registered skills (`claude -p /status` → `Unknown skill: status`). The "X% of weekly limit" number in the UI is computed server-side and is NOT written to any file under `~/.claude/` — `stats-cache.json` has only message/session/tool counts, transcript JSONL has per-message `usage` but no weekly %, and no local HTTP/socket API exposes it. For weekly/historical token usage, walk `~/.claude/projects/*/<session>.jsonl` `message.usage` fields; for the reported % itself, the user must read it off the UI manually.
- **[gotcha]** Claude Code transcripts replicate: the same assistant turn is replayed into every transcript that forks/resumes from the originating session — the same `message.id` can appear in 10-30 `.jsonl` files under `~/.claude/projects/`. Any walker that aggregates `message.usage` MUST dedup on `message.id + message.model + requestId` (same key `ccusage` uses); fall back to `timestamp + model + input_tokens + output_tokens` for pre-IDs transcripts. Missing dedup inflates token counts 2-3× and produces implausibly large weekly totals.
- **[gotcha]** Claude Code pre-warms every registered subagent at session start with a `"Warmup"` user message on a sidechain — each counts as a billable assistant turn. On days with many installed plugin agents (e.g. code-analysis-*, detective, etc.), >99% of daily input-token counts come from sidechain turns, not real work. When attributing token spend, split on `entry.isSidechain` (and optionally drop turns whose paired user message text is `"Warmup"`) or the numbers will mislead capacity planning.
- **[env]** To suppress the subagent Warmup drain, set `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` in the `env` block of `~/.claude/settings.json`. All-or-nothing: it also disables the Ctrl+B shortcut, the Bash tool's `run_in_background: true` parameter, and auto-backgrounding of long Bash tools — applies to both interactive TUI and `claude -p`. `CLAUDE_AGENT_SDK_DISABLE_BUILTIN_AGENTS=1` is a separate, narrower env var that only affects built-in subagents (Explore, Plan, etc.) and only in non-interactive `-p` mode. There is no per-agent or per-plugin opt-out. To keep a plugin's skills but drop its agents, disable the plugin (`claude plugin` → disabled) after running `extract_plugin.py` to copy the SKILL.md files into `~/.claude/skills/`.
- **[pattern]** Running PowerShell on a Windows SSH target via one-liner (`ssh host "powershell -Command ..."`) is extremely fragile — `$` and quote escaping silently corrupts `$s.env` into `\.env` and similar parse errors. Preferred approach: `Write` the script locally to a `.ps1`, `scp` it to the remote home dir (relative path, e.g. `pandorum:disable_warmup.ps1` — NOT `pandorum:/c/Users/...` which hits OpenSSH path translation and fails with `No such file or directory`), then `ssh host "powershell -NoProfile -ExecutionPolicy Bypass -File script.ps1"` and clean up with `ssh host "del script.ps1"`.
- **[correction]** When posting GitHub issue comments that link to scripts in the *current* repo, verify `git remote get-url origin` and use absolute `https://github.com/<owner>/<repo>/blob/main/...` URLs. Relative links like `../scripts/foo.py` resolve against the issue repo (e.g. `anthropics/claude-code`) — not your repo — and yield a broken path. This project's origin is `mann1x/claude-hooks`, not anything under the `anthropics/` org.
- **[pattern]** To edit an already-posted GitHub issue comment, use `gh api -X PATCH repos/<owner>/<repo>/issues/comments/<comment_id> -F body=@/path/to/file.md`. The comment ID is the trailing number on the `#issuecomment-NNNN` URL returned by `gh issue comment`. Do NOT re-post and delete — PATCH preserves the comment's created-at timestamp and notification thread.
- **[pattern]** `gh search issues ... --json state` returns `"state": "open"` / `"closed"` but omits the *reason* a closure happened. To distinguish "fix landed" from "won't fix / not planned", fetch the specific issue with `gh issue view <n> --json state,stateReason,closedAt` — `stateReason: "NOT_PLANNED"` means the bug is still live and a new issue is warranted. Always check before deciding whether to file a duplicate.
- **[gotcha]** Claude Code transcripts replicate: the same assistant turn is replayed into every transcript that forks/resumes from the originating session — the same `message.id` can appear in 10-30 `.jsonl` files under `~/.claude/projects/`. Any walker that aggregates `message.usage` MUST dedup on `message.id + message.model + requestId` (same key `ccusage` uses); fall back to `timestamp + model + input_tokens + output_tokens` for pre-IDs transcripts. Missing dedup inflates token counts 2-3× and produces implausibly large weekly totals. Validate by comparing to `message.id`-only dedup (should yield identical counts — `message.id` is already unique per API call) and cross-check against `ccusage daily -z <tz>` (deltas < 1 % are calendar-day-vs-reset-window drift, not a bug).
- **[correction]** When documenting third-party env var / workaround recommendations harvested from a big GitHub thread, do NOT present them as project-endorsed. User preference: each entry must carry a **field-tested verdict** (✅ recommended / ❌ not recommended / ⚠️ situational / ⚠️ untested) based on actual use on this project. Specifically: the "bcherny stack" (`CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1` + `MAX_THINKING_TOKENS=63999`) is marked ❌ — it introduced more trivial mistakes and slower iteration here. Pattern applies to any future `docs/env-vars.md`-style reference page.
- **[convention]** Installer prompts in `install.py` that inject values into `~/.claude/settings.json` (env vars, MCP servers, permissions) MUST default to No and require explicit `y` to apply. Back up `settings.json` (`backup_path()` + `shutil.copy`) before writing. Rationale: these changes are user-visible and affect every subsequent Claude Code session — silent opt-ins violate reversibility. User explicitly requested default-No for the bcherny-stack env-var injection prompt.
- **[pattern]** JSON config files (no comment support) can carry inline documentation via a top-level `"_doc_<topic>"` key whose value is an array of strings (one per line). `_`-prefixed keys are ignored by the loader but `python3 -c "import json; json.load(...)"` still validates. Used in `config/claude-hooks.example.json` to document Claude Code `env` block recommendations without a parser hack.
- **[gotcha]** Repos referenced in GitHub issue comments may not exist — e.g. `MadAppGang/claude-mem` returned 404 despite being mentioned. Before analysing or recommending a tool, verify existence with `curl -sL https://api.github.com/repos/<owner>/<name>` (or `WebFetch` — it returns an empty object `{}` on 404 rather than raising). Confusingly similar names across authors (`thedotmack/claude-mem` vs `fastmcp-me/mnemex` vs non-existent `MadAppGang/claude-mem` vs the `claudemem` CLI plugin) are common in this ecosystem.
- **[pattern]** In-process integration tests for an HTTP server: bind a fake upstream via `HTTPServer(("127.0.0.1", 0), Handler)` in a daemon thread, grab the OS-assigned port via `getsockname()[1]`, pass it as the proxy's `upstream`, and tear down with `server.shutdown(); server.server_close()` in a finally block. Avoids flaky tests from hard-coded ports and races. Used across `tests/test_proxy.py` and `tests/test_proxy_p1.py`.
- **[gotcha]** `http.client` normalises request header names case (e.g. `x-api-key` may arrive upstream as `X-Api-Key`). Proxy/forwarder tests asserting on preserved auth headers must normalise via `{k.lower(): v for k, v in captured.items()}` before lookup — direct `dict["x-api-key"]` access raises `KeyError`.
- **[gotcha]** Non-greedy tag-stripping regexes (e.g. `<tag>.*?</tag>` with `DOTALL`) do NOT handle nested same-name tags in a single `sub()` pass — an inner `<tag>…</tag>` pairs the outer opener with the inner closer and leaves the outer closer as an orphan. Fix: loop `re.sub` until the output stops changing (bound at 5 passes), then run an orphan-tag sweep `re.sub(rf"</?{tags}\b[^>]*>", "", …)` to clean up anything left. See `claude_hooks/hooks/stop.py::_strip_system_tags` for the pattern.
- **[gotcha]** Binding the proxy to a specific LAN IP (`listen_host: "192.168.178.2"`) means `curl http://127.0.0.1:38080/` returns `000` (connection refused) — only the LAN IP accepts connections. When smoke-testing, curl the actual bind address (`http://192.168.178.2:38080/`); a `404` from the upstream forwarder confirms the proxy is up and reachable. Use `0.0.0.0` if you need both local and LAN access.
