# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** Anthropic's edge rate-limits HTTP/1.1-per-request clients with **real 429s** even when `anthropic-ratelimit-unified-status=allowed` and 5h/7d budgets are well under quota. Observed: 4×429 in 58s on single non-overlapping ~2MB requests, concurrent=0. Root cause is the connection profile (fresh TCP+TLS per request) not request volume. The 429s carry **no** `anthropic-ratelimit-unified-*` headers — they come from a separate edge gate invisible to the published rate-limit API. Any forwarder in front of `api.anthropic.com` MUST use HTTP/2 with connection pooling (e.g. `httpx.Client(http2=True)`) to match the native Claude Code (Node undici) client profile.
- **[pattern]** When wiring a pooled HTTP client (`httpx.Client`) into a module that was previously stdlib-only, set `trust_env=False` on the client. Rationale: if this module IS the proxy the user configures via `HTTPS_PROXY` / `ANTHROPIC_BASE_URL`, trusting env-vars causes the client to route its own upstream calls back through itself → infinite loop / stall. Also use a module-level `atexit.register(_reset_client)` + `threading.Lock` guarded lazy init so tests can call `_reset_client()` between cases without leaking h2 connections.
- **[gotcha]** HTTP/2 normalises response header names to lowercase, so code that copies `resp.headers` into a plain dict loses the `Content-Type` casing it had under HTTP/1.1. Downstream lookups like `response_headers.get("Content-Type", "")` silently return `""` and SSE detection fails (the tailer is never attached, `result.sse_tail` is `None`). Either use `httpx.Headers.get("content-type")` (case-insensitive) directly off `resp.headers` before the dict copy, or always lower() lookup keys when reading the copied dict.
- **[gotcha]** `config/claude-hooks.json` is gitignored — running `git add config/claude-hooks.json` prints a hint (`The following paths are ignored by one of your .gitignore files`) and stages nothing. The file is user-local (contains real host IPs, MCP URLs, `proxy.enabled` state). Changes to defaults belong in `config/claude-hooks.example.json` (committed) and/or `DEFAULT_CONFIG` in `claude_hooks/config.py`; never try `git add -f` on the real config.
- **[env]** On pandorum, the claude-hooks conda env lives under `C:\Users\manni\Miniconda3\envs\claude-hooks` (NOT `anaconda3` — that path doesn't exist). Other envs present: `delsig`, `foo_dsd_trellis`, `lightseek`, `vodi`. Use `ssh pandorum 'where conda'` / `ssh pandorum 'conda env list'` to resolve the actual prefix before invoking pip/python by absolute path.
- **[gotcha]** `claude -p /cost` is non-functional for Claude subscription users — it only prints `"You are currently using your subscription to power your Claude Code usage"` and exits with no token data. `/status` and `/stats` are not registered skills (`claude -p /status` → `Unknown skill: status`). The "X% of weekly limit" number in the UI is computed server-side and is NOT written to any file under `~/.claude/` — `stats-cache.json` has only message/session/tool counts, transcript JSONL has per-message `usage` but no weekly %, and no local HTTP/socket API exposes it. For weekly/historical token usage, walk `~/.claude/projects/*/<session>.jsonl` `message.usage` fields; for the reported % itself, the user must read it off the UI manually — OR run the local proxy and read `ratelimit-state.json` which captures the real `anthropic-ratelimit-unified-*` headers.
- **[env]** `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` is **destructive** to interactive workflows — it kills any running background task (e.g. long-running model training) when the session that launched it exits. User ruled this env var out entirely: "it completely destroys claude code". Do NOT recommend it even as a Warmup-drain mitigation; the proxy-based Warmup block (`proxy.block_warmup: true`) is the only acceptable path. Keep this env var out of installer prompts, docs recommendations, and `config/claude-hooks.example.json`. `CLAUDE_AGENT_SDK_DISABLE_BUILTIN_AGENTS=1` is a separate, narrower env var that only affects built-in subagents (Explore, Plan, etc.) and only in non-interactive `-p` mode. There is no per-agent or per-plugin opt-out. To keep a plugin's skills but drop its agents, disable the plugin (`claude plugin` → disabled) after running `extract_plugin.py` to copy the SKILL.md files into `~/.claude/skills/`.
- **[pattern]** Running PowerShell on a Windows SSH target via one-liner (`ssh host "powershell -Command ..."`) is extremely fragile — `$` and quote escaping silently corrupts `$s.env` into `\.env` and similar parse errors, and multi-level quote nesting often returns empty stdout+stderr (silent failure) rather than an error. Preferred approach: `Write` the script locally to a `.ps1`, `scp` it to the remote home dir (relative path, e.g. `pandorum:enable_proxy.ps1` — NOT `pandorum:/c/Users/...` which hits OpenSSH path translation and fails with `No such file or directory`), then invoke with an **absolute** Windows path via `ssh host 'powershell -NoProfile -ExecutionPolicy Bypass -File C:\Users\manni\enable_proxy.ps1'`. Passing `$env:USERPROFILE\script.ps1` to `-File` fails with `The given path's format is not supported` — PowerShell's `-File` arg does NOT expand env vars. Clean up with a separate `ssh host "del script.ps1"` — NEVER chain cleanup after `-File` with `;` in the same SSH command, the shell passes the whole tail (`script.ps1; del script.ps1`) as the `-File` argument and PowerShell errors with `Processing -File 'script.ps1;' failed because the file does not have a '.ps1' extension.`
- **[pattern]** `gh search issues ... --json state` returns `"state": "open"` / `"closed"` but omits the *reason* a closure happened. To distinguish "fix landed" from "won't fix / not planned", fetch the specific issue with `gh issue view <n> --json state,stateReason,closedAt` — `stateReason: "NOT_PLANNED"` means the bug is still live and a new issue is warranted. Always check before deciding whether to file a duplicate. Also: `gh search issues --state` only accepts `{open|closed}` (not `all`), and `--json` does NOT accept the field name `comments` — use `commentsCount` instead; the error message lists all valid fields.
- **[correction]** When a third party opens a feature-request / integration issue on this repo that overlaps existing functionality, the user's preferred response pattern is: (1) state candid capacity status (weekly limit %, travel, etc.), (2) describe what this repo already does in the adjacent space with pointers to exact module paths, (3) call out *why* it's genuinely different (architecture, not just implementation), (4) list 2-3 concrete integration seams without committing, (5) ask the requester to post a written integration plan for async review. Do NOT be dismissive — the user explicitly overrode a colder draft with "I don't want to be too cold... no harm in supporting him."
- **[pattern]** Proxy smoke-test after restart: `curl -s -o /dev/null -w "status=%{http_code} proto=%{http_version} time=%{time_total}s\n" http://<bind>:<port>/v1/messages -X POST -H "x-api-key: bogus" …` — a `401` response with `proto=1.0` (curl's client side is HTTP/1.0, the proxy upstream is still HTTP/2) confirms end-to-end routing. To verify the **upstream** http_version, call `claude_hooks.proxy.forwarder.forward()` in-process and read `result.stats['http_version']` — should be `HTTP/2` after the httpx migration.
- **[gotcha]** `scp local pandorum:/c/Users/manni/file` (OpenSSH-style absolute Windows path with leading slash) fails silently — empty response, no error. Use a relative path (`pandorum:file.ps1`, lands in `%USERPROFILE%`) for scp, then reference the absolute path `C:\Users\manni\file.ps1` only on the ssh-side invocation.
- **[pattern]** To sync an uncommitted-config host (e.g. pandorum) to a new release: (1) `ssh host 'cd claude-hooks; git pull'` and `git log --oneline -3` to verify, (2) resolve the local conda env prefix via `ssh host 'conda env list'` (DO NOT assume `anaconda3/` vs `Miniconda3/`), (3) install new Python deps with the absolute `pip.exe` path from that prefix, (4) wire settings.json via a `.ps1` script using `ConvertFrom-Json` / `Add-Member -Force` / `ConvertTo-Json -Depth 20` and a `.bak` copy before write.
- **[gotcha]** `caliber status` reports `"configured": true` even when `manifest` is `null` and no init was actually completed — the presence of `.caliber/` dir and a global `~/.caliber/config.json` is enough. To verify a project was properly initialized, check `manifest` is non-null in `caliber status --json`. All 8 projects on solidPC and all 4 on pandorum showed `manifest=NULL` despite `.caliber/` dirs existing.
- **[gotcha]** `caliber regenerate` fails with "No existing config found. Run caliber init first." when manifest is null — you must run `caliber init` (not regenerate) to bootstrap from scratch. On this project, `caliber init --dry-run --auto-approve` timed out after ~2.5 min (`timeout_inactivity`) consuming ~870 KB request to Opus, pushing 5h utilization from 18% to 23%. Budget ~5% of 5h window per `caliber init` run on a large project.
- **[gotcha]** The proxy's systemd unit has `PrivateTmp=true` (Debian default for `DynamicUser` or explicit config). Files written to `/tmp/` from within the service are in a private mount namespace — they are invisible to processes outside the unit (including the Claude Code session). When debugging with file dumps from a systemd service, write to a path outside `/tmp` (e.g. `/root/.claude/claude-hooks-proxy-dumps/`) or the files will appear empty/missing.
- **[gotcha]** `httpx` `resp.iter_raw()` yields transport-level bytes (possibly gzip/br compressed). When Anthropic's upstream occasionally applies content-encoding to SSE responses, the raw bytes contain no `\n\n` delimiters and `SseTail` can't parse any events — `stop_reason`, `usage`, and thinking metrics all come back `None`. Use `resp.iter_bytes()` (content-decoded) instead. The downstream client (`wfile.write()`) receives decoded bytes either way because httpx strips `Content-Encoding` from response headers when decoding.
- **[gotcha]** Test fixtures that share a default timestamp (e.g. `_mk_record()` always returns `ts="2026-04-14T10:00:00Z"`) silently collide on `UNIQUE(source_file, source_line)` constraints in stats_db — the second record is `INSERT OR IGNORE`'d away. Always give each record in a multi-record test a distinct `ts` value to avoid phantom undercounts.
- **[convention]** Installer prompts in `install.py` that inject values into `~/.claude/settings.json` (env vars, MCP servers, permissions) MUST default to No and require explicit `y` to apply. Back up `settings.json` (`backup_path()` + `shutil.copy`) before writing. Rationale: these changes are user-visible and affect every subsequent Claude Code session — silent opt-ins violate reversibility. User explicitly requested default-No for the bcherny-stack env-var injection prompt.
- **[pattern]** JSON config files (no comment support) can carry inline documentation via a top-level `"_doc_<topic>"` key whose value is an array of strings (one per line). `_`-prefixed keys are ignored by the loader but `python3 -c "import json; json.load(...)"` still validates. Used in `config/claude-hooks.example.json` to document Claude Code `env` block recommendations without a parser hack.
- **[gotcha]** Repos referenced in GitHub issue comments may not exist — e.g. `MadAppGang/claude-mem` returned 404 despite being mentioned. Before analysing or recommending a tool, verify existence with `curl -sL https://api.github.com/repos/<owner>/<name>` (or `WebFetch` — it returns an empty object `{}` on 404 rather than raising). Confusingly similar names across authors (`thedotmack/claude-mem` vs `fastmcp-me/mnemex` vs non-existent `MadAppGang/claude-mem` vs the `claudemem` CLI plugin) are common in this ecosystem.
- **[gotcha]** When a systemd unit has `PrivateTmp=yes` (the default for many service templates), files written to `/tmp` inside the service are invisible from the host shell — they live in a private mount namespace under `/tmp/systemd-private-*/`. Use a path outside `/tmp` (e.g. under the service's own data directory or `~/.claude/`) for debug output that needs to be readable from outside the service. Symptom: the directory appears empty from the host even though the service logged successful writes to it.
- **[pattern]** In-process integration tests for an HTTP server: bind a fake upstream via `HTTPServer(("127.0.0.1", 0), Handler)` in a daemon thread, grab the OS-assigned port via `getsockname()[1]`, pass it as the proxy's `upstream`, and tear down with `server.shutdown(); server.server_close()` in a finally block. Avoids flaky tests from hard-coded ports and races. Used across `tests/test_proxy.py` and `tests/test_proxy_p1.py`.
- **[gotcha]** `http.client` normalises request header names case (e.g. `x-api-key` may arrive upstream as `X-Api-Key`). Proxy/forwarder tests asserting on preserved auth headers must normalise via `{k.lower(): v for k, v in captured.items()}` before lookup — direct `dict["x-api-key"]` access raises `KeyError`.
- **[gotcha]** Non-greedy tag-stripping regexes (e.g. `<tag>.*?</tag>` with `DOTALL`) do NOT handle nested same-name tags in a single `sub()` pass — an inner `<tag>…</tag>` pairs the outer opener with the inner closer and leaves the outer closer as an orphan. Fix: loop `re.sub` until the output stops changing (bound at 5 passes), then run an orphan-tag sweep `re.sub(rf"</?{tags}\b[^>]*>", "", …)` to clean up anything left. See `claude_hooks/hooks/stop.py::_strip_system_tags` for the pattern.
- **[gotcha]** Binding the proxy to a specific LAN IP (`listen_host: "192.168.178.2"`) means `curl http://127.0.0.1:38080/` returns `000` (connection refused) — only the LAN IP accepts connections. When smoke-testing, curl the actual bind address (`http://192.168.178.2:38080/`); a `404` from the upstream forwarder confirms the proxy is up and reachable. Use `0.0.0.0` if you need both local and LAN access.
- **[gotcha]** systemd's `PrivateTmp=true` (default on many unit files) creates an isolated `/tmp` namespace per service — files the service writes to `/tmp/` are invisible from the host and from other services. If you need a proxy dump dir or debug output accessible from an interactive shell, write to a path outside `/tmp/` (e.g. `/root/.claude/claude-hooks-proxy-dumps/`) or use a systemd drop-in to set the env var. Symptom: `ls /tmp/proxy-dump-s2/` shows an empty dir from the host shell even though journal logs confirm writes succeeded.
- **[gotcha]** The proxy's `handler.path` includes query parameters (e.g. `/v1/messages?beta=true`), so path matching like `handler.path != "/v1/messages"` silently fails when the client appends query strings. Always strip the query first: `handler.path.split("?", 1)[0]`.
- **[gotcha]** The proxy forwarder MUST strip `Accept-Encoding` from outbound request headers and pin `Accept-Encoding: identity`. Without this, Anthropic's upstream may apply gzip/br content-encoding to SSE responses, and `httpx` `resp.iter_raw()` yields compressed bytes with no `\n\n` delimiters — `SseTail` can't parse any events, so `stop_reason`, `usage`, thinking metrics, and all SSE-derived fields come back `None`. Stripping the encoding offer is safer than switching to `iter_bytes()` because it keeps the transport transparent and avoids double-decompression edge cases. The bandwidth cost over localhost is negligible.
- **[gotcha]** Claude Opus 4.6 with `redact-thinking-2026-02-12` beta streams thinking blocks as `content_block_start` with `type: "thinking"` and an empty `signature: ""`, followed by `content_block_delta` events with `delta.type: "signature_delta"` (NOT `"thinking_delta"`). The `signature` field on each delta carries base64-encoded bytes that accumulate the thinking signature. Match on `"thinking" in block_type` for content_block_start and `delta_type == "signature_delta" or "thinking" in delta_type` for deltas to handle both redacted and non-redacted variants.
