# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[convention]** When extending the `stop_guard` (or any pattern-matching hook) with new behaviour, also test the *meta-context escape*: a message that quotes or documents a trigger phrase (`"pre-existing issue"`) must NOT fire the hook. The `MetaContextEscapeTests` class in `tests/test_stop_guard.py` is the reference pattern — reuse it for similar hooks.
- **[gotcha]** `claudemem <subcmd> --help` and `claudemem --agent <subcmd> --help` both just re-print the global help — they do NOT show subcommand-specific usage. Use `claudemem <subcmd> help` (no dashes) instead, e.g. `claudemem hooks help` prints the subcommands for `hooks`. `claudemem --agent watch` additionally spawns a long-running daemon; if run unintentionally, kill with `pkill -f 'claudemem.*watch'`.
- **[pattern]** `claudemem hooks install` sets up a post-commit git hook at `.git/hooks/post-commit` that runs `claudemem index --quiet` in the background after every commit. It's per-project — install it once per repo. Verify with `claudemem hooks status`. For claude-hooks, this covers committed changes; mid-session uncommitted edits are handled by the `claudemem_reindex` hook in this project.
- **[pattern]** User's terse prompts ("delete it", "revert it", single-word answers) are frequently symptoms of Claude Code TUI input lag — keyboard echo is slowed and chars get dropped, so the user truncates mid-sentence. Default to asking "which one?" when pronouns point to multiple candidates; do NOT optimise for fewer round-trips at the cost of guessing wrong. Per-turn hook latency also compounds the client lag, so be conservative when adding new hook work.
- **[correction]** Do NOT probe Ollama model availability via `https://registry.ollama.ai/v2/library/<name>/tags/list` — that path returns 404 for every repo and is not a valid public endpoint. To verify a tag exists, fetch the manifest: `curl -sL -o /dev/null -w "%{http_code}" https://registry.ollama.ai/v2/library/<name>/manifests/<tag>` (expects 200), or load the library page `https://ollama.com/library/<name>`. Getting this wrong leads to false "model doesn't exist publicly" conclusions.
- **[gotcha]** In Claude Code PreToolUse hooks, returning an explicit `permissionDecision` (either `allow` or `ask`) BYPASSES the user's `~/.claude/settings.json` allow-list — the hook decision is authoritative. So when one stage emits `allow` (e.g. rtk auto-approving a rewrite), any dangerous-pattern scanner MUST run first on the effective command. Do not gate the scanner behind a separate enable flag if it's also the safety net for an auto-allow stage.
- **[pattern]** Keep `config/claude-hooks.example.json` in lockstep with `claude_hooks.config.DEFAULT_CONFIG` by regenerating it programmatically (`python3 -c "from claude_hooks.config import DEFAULT_CONFIG; json.dump(...)"`) with a small set of placeholder-URL substitutions (`http://YOUR-HOST:32775/mcp` etc.), rather than hand-editing. Manual edits silently drift — the example missed several keys until regenerated.
- **[gotcha]** Claudemem's primary index store is `.claudemem/index.db`, NOT `.claudemem/index.bin`. When checking index freshness, stat `index.db` first and fall back to the directory mtime — do not walk the whole `.claudemem/` tree. Tests that write a fake index file must use `index.db`.
- **[pattern]** `claudemem status` emits structured key=value lines (`exists=`, `files=`, `chunks=`, `languages=`, `model=`, `last_updated=`). Parse with shell/awk directly — no jq needed.
- **[correction]** "delete it" in this project means delete the *specific artefact just discussed* (e.g. the local backup file), NOT the whole feature/module that was the broader topic. When an instruction is ambiguous between a narrow and broad target, ask before deleting source files / tests / config blocks.
- **[convention]** Hooks that fire on every tool call or turn (safety_scan, stop_guard, rtk_rewrite) MUST cache compiled regex patterns and binary-version probes at module level, keyed by the config that produced them. Provide a `reset_*_cache()` helper for tests. Re-compiling ~40 regexes or re-invoking `rtk --version` on every PreToolUse event adds measurable per-turn latency that compounds across hooks.
- **[gotcha]** `.git` in a project root can be a regular directory (normal repo) OR a plain file (git worktree / submodule pointing at the real gitdir). Project-root detection must use `.exists()` not `.is_dir()`, otherwise worktrees are silently treated as non-repos. Applies to `claude_hooks.claudemem_reindex._project_root` and any similar helpers.
- **[gotcha]** `claude_hooks.hyde._call_ollama` is keyword-only (`def _call_ollama(*, user_prompt, system_prompt, model, url, timeout, max_tokens, keep_alive=...)`) AND silently returns `""` for any response shorter than 10 characters. Tests that mock the Ollama endpoint must supply a response ≥10 chars, not a short token like `"ok"`.
- **[gotcha]** Running `git commit && git push` in a Bash tool call with `run_in_background: true` (or when Claude backgrounds it automatically) while the main shell continues to run `git` commands produces `fatal: cannot lock ref 'HEAD': is at <new> but expected <old>`. The background push usually succeeded — diagnose with `git fetch && git log origin/main -3`; don't retry the commit or you'll double-apply. Recover with `git pull --ff-only` after verifying the remote already has the commit.
- **[gotcha]** `claude_hooks.hyde._format_context(mems, max_chars=N)` returns `""` (not a partial result) when the first formatted line `"1. <snippet>\n"` exceeds the total budget. The per-entry cap is `max(200, N // len(mems))` so single-item or giant-entry cases overflow easily — tests must set `max_chars` comfortably larger than `per_entry_cap + len("N. ") + 1`, or stick to small/numerous inputs.
- **[gotcha]** `claude_hooks.instincts.detect_bug_fix` only scans content blocks that appear AFTER the last `user`-role message in the transcript. Tool-result blocks placed on a user-role message are treated as the end-of-turn marker, not as error evidence. In tests, put the whole Bash → tool_result → Edit sequence on one or more **assistant**-role messages after the initial user prompt, not alternating user/assistant turns.
- **[pattern]** `stop_guard` has three escape layers, checked in this order: (1) user-wrap-up markers on the last USER message (`skip_on_user_wrap_up`, default on — e.g. "compact the context", "wrap up", "save state", "/wrapup") short-circuit before any pattern scan; (2) meta-marker phrases in the assistant text; (3) quoted-span stripping. When wiring the guard into a new handler, thread the last user-role text through `check_message(..., last_user_message=...)`; passing only the assistant text disables the strongest escape and causes false positives on legitimate wrap-up requests.
- **[convention]** Do not commit unverified test files. When the user interrupts mid-authoring (a `test_*.py` written but never run green), `rm` it before staging the rest of the phase — running `pytest tests/ -q` and getting `<N> passed` is the bar for inclusion. Commit messages that claim a phase is done must reflect only files whose tests are currently green.
- **[pattern]** `tests/mocks/ollama.py::mock_ollama_generate(target="claude_hooks.<module>")` patches `urllib.request.urlopen` *inside the named module*, not globally. The `target` argument must match the module whose `urlopen` call you want to intercept — use `claude_hooks.reflect` when testing `reflect.py`, `claude_hooks.consolidate` for `consolidate.py`, `claude_hooks.hyde` for HyDE. Using the wrong target silently passes the real urlopen through and lets tests hit the network.
- **[pattern]** Coverage gate runs via conda: `conda run -n claude-hooks coverage run -m pytest tests/` then `coverage report`. Config lives in `pyproject.toml` under `[tool.coverage.run]` and `[tool.coverage.report]` — branch coverage enabled, `pgvector.py`/`sqlite_vec.py`/`detect.py` omitted (integration / interactive code). Target ≥ 80 % branch. Commit `requirements-dev.txt` + `pyproject.toml` together when bumping coverage deps.
- **[gotcha]** When mocking `McpClient` in provider tests, patch `claude_hooks.mcp_client.McpClient` — NOT `claude_hooks.providers.base.McpClient`. Providers import the client lazily inside their `_client()` method, so the name does not exist in `providers.base`'s namespace and the patch raises `AttributeError: module ... does not have the attribute 'McpClient'`. Same rule applies to any symbol imported inside a method vs at module top level.
- **[gotcha]** `patch("builtins.__import__", side_effect=ImportError(...))` is far too broad — Python's `logging` module imports helpers internally on every `log.error()` call, so error-path tests that trigger a log will crash with a cascading ImportError. Use a selective side_effect that only raises for the specific module prefix under test (e.g. `if name.startswith("claude_hooks.hooks."): raise ImportError("boom"); return real_import(name, *a, **kw)`) and pass everything else through the real importer.
- **[gotcha]** `patch("sys.stdout")` with the default `MagicMock` breaks Python's logging: the log handler compares `record.levelno >= hdlr.level` where `hdlr.level` becomes a MagicMock, raising `TypeError: '>=' not supported between instances of 'int' and 'MagicMock'`. Replace stdout with a real minimal class that defines `write(self, s)` and `flush(self)` (and raises from `write` if you want to simulate failure) instead of a MagicMock.
- **[pattern]** Tests that mutate the `claude_hooks` logger (adding/removing handlers, changing level) need a save/restore fixture — otherwise state leaks and later tests see unexpected handlers or log output. Snapshot `root.handlers[:]` and `root.level` in setup, restore in teardown, and `.close()` any handlers the test added. Avoid patching `logging.handlers.RotatingFileHandler` without this fixture; the module's global state will persist across tests.
- **[correction]** Don't declare a coverage plan "complete" just because the configured minimum (≥ 80 %) was met — check the `coverage report --skip-covered` gap list and surface it. User preference: push coverage further when achievable even at diminishing returns ("I want a reasonably complete test coverage"). In this project 92 % was reachable in one focused phase targeting the lowest modules; stopping at 81 % when 92 % is a few hundred lines of boundary-mocking away under-delivers.
- **[convention]** When adding `coverage.py` to a project, also add `.coverage`, `.coverage.*`, and `htmlcov/` to `.gitignore` — coverage writes these to the repo root by default and they'll otherwise be listed as untracked on every run.
- **[gotcha]** `claude -p /cost` is non-functional for Claude subscription users — it only prints `"You are currently using your subscription to power your Claude Code usage"` and exits with no token data. `/cost` itself is session-scoped and API-user oriented. For weekly/historical usage, walk the transcript JSONL files under `~/.claude/projects/*/<session>.jsonl`: each `{"type":"assistant"}` entry carries `message.usage` with `input_tokens`, `output_tokens`, `cache_creation_input_tokens`, and `cache_read_input_tokens`. `~/.claude/stats-cache.json` has only `messageCount`/`sessionCount`/`toolCallCount` — no tokens.
- **[gotcha]** `ccusage` (`npx -y ccusage@latest`) groups usage by calendar day in whatever tz you pass via `-z` and cannot honour a non-midnight reset boundary (e.g. Claude's Friday 10:00 CEST weekly reset). Use it for USD pricing and per-model breakdowns (`daily --breakdown`), but roll your own transcript walker when you need the exact weekly-limit window.
- **[env]** On pandorum, the `claude-hooks` checkout is at `C:\Users\manni\claude-hooks` — NOT `C:\Users\manni\dev\claude-hooks`. Sync with `ssh pandorum "powershell -NoProfile -Command \"cd C:\Users\manni\claude-hooks; git pull --ff-only origin main\""`. Guessing `dev\claude-hooks` fails with `"The system cannot find the path specified."`
- **[gotcha]** `claude -p /cost` is non-functional for Claude subscription users — it only prints `"You are currently using your subscription to power your Claude Code usage"` and exits with no token data. `/status` and `/stats` are not registered skills (`claude -p /status` → `Unknown skill: status`). The "X% of weekly limit" number in the UI is computed server-side and is NOT written to any file under `~/.claude/` — `stats-cache.json` has only message/session/tool counts, transcript JSONL has per-message `usage` but no weekly %, and no local HTTP/socket API exposes it. For weekly/historical token usage, walk `~/.claude/projects/*/<session>.jsonl` `message.usage` fields; for the reported % itself, the user must read it off the UI manually.
