# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[env]** The claude-hooks conda env on solidPC lives at `/root/anaconda3/envs/claude-hooks/bin/python`. Do NOT rely on `source /opt/conda/etc/profile.d/conda.sh && conda activate claude-hooks` — it silently fails in non-interactive Bash tool invocations (returns empty stdout+stderr). Invoke the absolute interpreter path directly: `/root/anaconda3/envs/claude-hooks/bin/python -m pytest …`.
- **[gotcha]** When changing forwarder retry defaults, audit existing tests that rely on timing-out a connect attempt (e.g. `test_upstream_failure_yields_502` pointing at a dead port). With `_UPSTREAM_RETRIES=10` and the httpx default connect timeout, total latency exceeds most urllib client timeouts. Tests that exercise the connection-error path must monkeypatch `fwd._UPSTREAM_RETRIES` down to 1 AND `_RETRY_BACKOFF_BASE`/`_RETRY_BACKOFF_MAX` to 0, plus call `fwd._reset_client()` in teardown.
- **[pattern]** For a retryable-upstream-status path in the forwarder, the retry loop MUST buffer and preserve the *authentic* upstream response (status, headers stripped of hop-by-hop, body) so that — on retry exhaustion — the client sees the real error, not a proxy-synthesized `502 proxy_error` wrapper. Use a custom exception class (e.g. `_RetryableStatus`) carrying `(status, reason, body, headers)` and, after the loop, synthesize an `UpstreamResult` rather than re-raising.
- **[env]** On pandorum, the claude-hooks conda env lives under `C:\Users\manni\Miniconda3\envs\claude-hooks` (NOT `anaconda3` — that path doesn't exist). Other envs present: `delsig`, `foo_dsd_trellis`, `lightseek`, `vodi`. Use `ssh pandorum 'where conda'` / `ssh pandorum 'conda env list'` to resolve the actual prefix before invoking pip/python by absolute path.
- **[env]** `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` is **destructive** to interactive workflows — it kills any running background task when the session that launched it exits. User ruled this env var out entirely. Do NOT recommend it even as a Warmup-drain mitigation; the proxy-based Warmup block (`proxy.block_warmup: true`) is the only acceptable path. `CLAUDE_AGENT_SDK_DISABLE_BUILTIN_AGENTS=1` is a separate, narrower env var that only affects built-in subagents in non-interactive `-p` mode.
- **[pattern]** Running PowerShell on a Windows SSH target via one-liner (`ssh host "powershell -Command ..."`) is extremely fragile — `$` and quote escaping silently corrupts `$s.env` into `\.env` and multi-level quote nesting often returns empty stdout+stderr. Preferred approach: `Write` the script locally to a `.ps1`, `scp` it to the remote home dir (relative path like `pandorum:enable_proxy.ps1` — NOT `pandorum:/c/Users/...` which hits OpenSSH path translation and fails), then invoke with an **absolute** Windows path via `ssh host 'powershell -NoProfile -ExecutionPolicy Bypass -File C:\Users\manni\enable_proxy.ps1'`. Passing `$env:USERPROFILE\script.ps1` to `-File` fails — PowerShell's `-File` arg does NOT expand env vars. Clean up with a separate `ssh host "del script.ps1"` — NEVER chain cleanup after `-File` with `;` in the same SSH command.
- **[pattern]** `gh search issues ... --json state` returns `"state": "open"` / `"closed"` but omits the *reason* a closure happened. To distinguish "fix landed" from "won't fix / not planned", fetch the specific issue with `gh issue view <n> --json state,stateReason,closedAt` — `stateReason: "NOT_PLANNED"` means the bug is still live and a new issue is warranted. Also: `gh search issues --state` only accepts `{open|closed}` (not `all`), and `--json` does NOT accept the field name `comments` — use `commentsCount`.
- **[correction]** When a third party opens a feature-request / integration issue on this repo that overlaps existing functionality, the user's preferred response pattern is: (1) state candid capacity status, (2) describe what this repo already does with pointers to exact module paths, (3) call out *why* it's genuinely different (architecture, not just implementation), (4) list 2-3 concrete integration seams without committing, (5) ask the requester to post a written integration plan for async review. Do NOT be dismissive — the user explicitly overrode a colder draft with "I don't want to be too cold... no harm in supporting him."
- **[pattern]** Proxy smoke-test after restart: `curl -s -o /dev/null -w "status=%{http_code} proto=%{http_version} time=%{time_total}s\n" http://<bind>:<port>/v1/messages -X POST -H "x-api-key: bogus" …` — a `401` response with `proto=1.0` (curl's client side is HTTP/1.0, the proxy upstream is still HTTP/2) confirms end-to-end routing. To verify the **upstream** http_version, call `claude_hooks.proxy.forwarder.forward()` in-process and read `result.stats['http_version']` — should be `HTTP/2` after the httpx migration.
- **[gotcha]** `scp local pandorum:/c/Users/manni/file` (OpenSSH-style absolute Windows path with leading slash) fails silently — empty response, no error. Use a relative path (`pandorum:file.ps1`, lands in `%USERPROFILE%`) for scp, then reference the absolute path `C:\Users\manni\file.ps1` only on the ssh-side invocation.
- **[pattern]** To sync an uncommitted-config host (e.g. pandorum) to a new release: (1) `ssh host 'cd claude-hooks; git pull'` and `git log --oneline -3` to verify, (2) resolve the local conda env prefix via `ssh host 'conda env list'` (DO NOT assume `anaconda3/` vs `Miniconda3/`), (3) install new Python deps with the absolute `pip.exe` path from that prefix, (4) wire settings.json via a `.ps1` script using `ConvertFrom-Json` / `Add-Member -Force` / `ConvertTo-Json -Depth 20` and a `.bak` copy before write.
- **[gotcha]** `caliber regenerate` fails with "No existing config found. Run caliber init first." when manifest is null — you must run `caliber init` (not regenerate) to bootstrap from scratch. The regenerate/init check is `fs.existsSync(.caliber/manifest.json)` — without that file, only `init` works. Budget ~5% of 5h utilization window per `caliber init` run on a large project.
- **[gotcha]** Caliber has THREE independent timeouts that each kill a long generation; defaults are too aggressive for large projects. Set all three before `caliber refresh|regenerate|init`: `CALIBER_CLAUDE_CLI_TIMEOUT_MS=3600000` (overall claude CLI), `CALIBER_GENERATION_TIMEOUT_MS=1800000` (total generation step, default 600000), `CALIBER_STREAM_INACTIVITY_TIMEOUT_MS=600000` (no-output gap, default 120000). Read `<project>/.caliber/error-log.md` after a failure — it names the specific timeout that fired (e.g. `Stop reason: timeout_inactivity`).
- **[gotcha]** Caliber 1.45.2 `init` has a hook-recursion bug: it installs its own SessionEnd hooks (including `caliber learn finalize --auto`) *before* spawning `claude -p` for generation. The subprocess's SessionEnd then fires `caliber learn finalize --auto` which races with the parent and fails with `SessionEnd hook [/usr/local/bin/caliber learn finalize --auto] failed: Hook cancelled` → `Model responded but output was not...` → generation aborts. Stripping `.claude/settings.json` hooks manually does NOT help — init re-adds them via `installLearningHooks()` before spawning. `PATH`-shim wrapping claude with `--bare` bypasses the hook but also breaks OAuth (login check uses the same `claude auth status` command). Workarounds: downgrade to 1.42.0, edit CLAUDE.md manually for grounding boosts, or use `caliber refresh` (incremental doc update) which doesn't trigger the generate loop.
- **[gotcha]** The proxy's systemd unit has `PrivateTmp=true` (Debian default). Files written to `/tmp/` from within the service are in a private mount namespace (`/tmp/systemd-private-*/`) — invisible from the host shell and other services. For proxy dump dirs / debug output that need to be readable from outside the service, use a path outside `/tmp/` (e.g. `/root/.claude/claude-hooks-proxy-dumps/`). Symptom: directory appears empty from the host even though journal logs confirm writes.
- **[gotcha]** `httpx` `resp.iter_raw()` yields transport-level bytes (possibly gzip/br compressed). When Anthropic's upstream applies content-encoding to SSE responses, the raw bytes contain no `\n\n` delimiters and `SseTail` can't parse any events — `stop_reason`, `usage`, thinking metrics all come back `None`. Fix by stripping `Accept-Encoding` from outbound request headers and pinning `Accept-Encoding: identity`.
- **[gotcha]** Test fixtures that share a default timestamp (e.g. `_mk_record()` always returning `ts="2026-04-14T10:00:00Z"`) silently collide on `UNIQUE(source_file, source_line)` constraints in stats_db — the second record is `INSERT OR IGNORE`'d away. Always give each record in a multi-record test a distinct `ts` value to avoid phantom undercounts.
- **[convention]** Installer prompts in `install.py` that inject values into `~/.claude/settings.json` (env vars, MCP servers, permissions) MUST default to No and require explicit `y` to apply. Back up `settings.json` (`backup_path()` + `shutil.copy`) before writing.
- **[pattern]** JSON config files (no comment support) can carry inline documentation via a top-level `"_doc_<topic>"` key whose value is an array of strings (one per line). `_`-prefixed keys are ignored by the loader but `python3 -c "import json; json.load(...)"` still validates.
- **[gotcha]** Repos referenced in GitHub issue comments may not exist — e.g. `MadAppGang/claude-mem` returned 404 despite being mentioned. Before analysing or recommending a tool, verify existence with `curl -sL https://api.github.com/repos/<owner>/<name>` (or `WebFetch` — it returns an empty object `{}` on 404 rather than raising).
- **[pattern]** In-process integration tests for an HTTP server: bind a fake upstream via `HTTPServer(("127.0.0.1", 0), Handler)` in a daemon thread, grab the OS-assigned port via `getsockname()[1]`, pass it as the proxy's `upstream`, and tear down with `server.shutdown(); server.server_close()` in a finally block. Used across `tests/test_proxy.py` and `tests/test_proxy_p1.py`.
- **[pattern]** For flaky-upstream tests in the forwarder, build a helper `_build_flaky_server(responses)` that returns each `(status, body)` in order then falls through to 200 — lets a single helper cover "502-then-200", "N consecutive 5xx", "status-list override" cases without duplicating server boilerplate. See `TestForwarderStatusRetry` in `tests/test_proxy_coverage.py`.
- **[gotcha]** Non-greedy tag-stripping regexes (e.g. `<tag>.*?</tag>` with `DOTALL`) do NOT handle nested same-name tags in a single `sub()` pass. Fix: loop `re.sub` until the output stops changing (bound at 5 passes), then run an orphan-tag sweep `re.sub(rf"</?{tags}\b[^>]*>", "", …)`. See `claude_hooks/hooks/stop.py::_strip_system_tags`.
- **[gotcha]** Binding the proxy to a specific LAN IP (`listen_host: "192.168.178.2"`) means `curl http://127.0.0.1:38080/` returns `000` (connection refused) — only the LAN IP accepts connections. When smoke-testing, curl the actual bind address; a `404` from the upstream forwarder confirms the proxy is up and reachable. Use `0.0.0.0` if you need both local and LAN access.
- **[gotcha]** The proxy's `handler.path` includes query parameters (e.g. `/v1/messages?beta=true`), so path matching like `handler.path != "/v1/messages"` silently fails when the client appends query strings. Always strip the query first: `handler.path.split("?", 1)[0]`.
- **[gotcha]** Claude Opus 4.6 with `redact-thinking-2026-02-12` beta streams thinking blocks as `content_block_start` with `type: "thinking"` and empty `signature: ""`, followed by `content_block_delta` events with `delta.type: "signature_delta"` (NOT `"thinking_delta"`). Match on `"thinking" in block_type` for content_block_start and `delta_type == "signature_delta" or "thinking" in delta_type` for deltas.
- **[gotcha]** The proxy forwards ALL upstream traffic including Claude Code's telemetry (`POST /api/event_logging/batch`) and health probes (`HEAD /`), which have NULL `model_effective`. Any rollup or dashboard query that aggregates by model must filter to `path = '/v1/messages'` — otherwise NULL-model rows pile up as a confusing `<unknown>` bucket.
- **[gotcha]** `caliber learn finalize --auto` silently skips analysis when events < 25 (no stderr, exit 0, empty finalize.log). The 25-event threshold also applies to non-auto runs and `--force` does NOT lower it — it only bypasses the running-process check. `.caliber/learning/finalize.lock` contains the PID; check `ps -fp $(cat .caliber/learning/finalize.lock)` before assuming stale — a Claude-Code-hook-triggered finalize can legitimately run many minutes in background.
- **[pattern]** When adjusting forwarder retry behaviour (e.g. bumping `_UPSTREAM_RETRIES`), existing failure-path tests that dial a dead TCP port will time out because each attempt pays the full connect-timeout. Fix the test with `monkeypatch.setattr(fwd, "_UPSTREAM_RETRIES", 1)` + zero backoff + `fwd._reset_client()` in teardown — do NOT shrink the production default to satisfy tests. See `tests/test_proxy.py::test_upstream_failure_yields_502`.
- **[gotcha]** `tests/test_dashboard.py::TestEndpoints::test_summary_today_populated` is a pre-existing flake unrelated to local changes: it hard-codes `"2026-04-15"` and now fails with `TypeError: 'NoneType' object is not subscriptable` on HEAD as well. When triaging test failures, stash changes and re-run the failing test against HEAD before assuming your edit caused it.
