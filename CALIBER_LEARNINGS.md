# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** When bulk-deploying a new pre-commit hook to multiple pandorum repos, back up the existing hook first (`cp ... .git/hooks/pre-commit.bak-YYYYMMDD`) before `scp`. Verify deploy with `wc -l` (expected line count) rather than `file`/`head` — Windows git doesn't care about the +x bit so chmod is a no-op there, but size/line count cheaply confirms the copy.
- **[env]** The claude-hooks conda env on solidPC lives at `/root/anaconda3/envs/claude-hooks/bin/python`. Do NOT rely on `source /opt/conda/etc/profile.d/conda.sh && conda activate claude-hooks` — it silently fails in non-interactive Bash tool invocations (returns empty stdout+stderr). Invoke the absolute interpreter path directly: `/root/anaconda3/envs/claude-hooks/bin/python -m pytest …`.
- **[gotcha]** When changing forwarder retry defaults, audit existing tests that rely on timing-out a connect attempt (e.g. `test_upstream_failure_yields_502` pointing at a dead port). With `_UPSTREAM_RETRIES=10` and the httpx default connect timeout, total latency exceeds most urllib client timeouts. Tests that exercise the connection-error path must monkeypatch `fwd._UPSTREAM_RETRIES` down to 1 AND `_RETRY_BACKOFF_BASE`/`_RETRY_BACKOFF_MAX` to 0, plus call `fwd._reset_client()` in teardown.
- **[pattern]** For a retryable-upstream-status path in the forwarder, the retry loop MUST buffer and preserve the *authentic* upstream response (status, headers stripped of hop-by-hop, body) so that — on retry exhaustion — the client sees the real error, not a proxy-synthesized `502 proxy_error` wrapper. Use a custom exception class (e.g. `_RetryableStatus`) carrying `(status, reason, body, headers)` and, after the loop, synthesize an `UpstreamResult` rather than re-raising.
- **[env]** On pandorum, the claude-hooks conda env lives under `C:\Users\manni\Miniconda3\envs\claude-hooks` (NOT `anaconda3` — that path doesn't exist). Other envs present: `delsig`, `foo_dsd_trellis`, `lightseek`, `vodi`. Use `ssh pandorum 'where conda'` / `ssh pandorum 'conda env list'` to resolve the actual prefix before invoking pip/python by absolute path.
- **[env]** `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1` is **destructive** to interactive workflows — it kills any running background task when the session that launched it exits. User ruled this env var out entirely. Do NOT recommend it even as a Warmup-drain mitigation; the proxy-based Warmup block (`proxy.block_warmup: true`) is the only acceptable path. `CLAUDE_AGENT_SDK_DISABLE_BUILTIN_AGENTS=1` is a separate, narrower env var that only affects built-in subagents in non-interactive `-p` mode.
- **[pattern]** Running PowerShell on a Windows SSH target via one-liner (`ssh host "powershell -Command ..."`) is extremely fragile — `$` and quote escaping silently corrupts `$s.env` into `\.env` and multi-level quote nesting often returns empty stdout+stderr. Preferred approach: `Write` the script locally to a `.ps1`, `scp` it to the remote home dir (relative path like `pandorum:enable_proxy.ps1` — NOT `pandorum:/c/Users/...` which hits OpenSSH path translation and fails), then invoke with an **absolute** Windows path via `ssh host 'powershell -NoProfile -ExecutionPolicy Bypass -File C:\Users\manni\enable_proxy.ps1'`. Passing `$env:USERPROFILE\script.ps1` to `-File` fails — PowerShell's `-File` arg does NOT expand env vars. Clean up with a separate `ssh host "del script.ps1"` — NEVER chain cleanup after `-File` with `;` in the same SSH command.
- **[pattern]** `gh search issues ... --json state` returns `"state": "open"` / `"closed"` but omits the *reason* a closure happened. To distinguish "fix landed" from "won't fix / not planned", fetch the specific issue with `gh issue view <n> --json state,stateReason,closedAt` — `stateReason: "NOT_PLANNED"` means the bug is still live and a new issue is warranted. Also: `gh search issues --state` only accepts `{open|closed}` (not `all`), and `--json` does NOT accept the field name `comments` — use `commentsCount`.
- **[correction]** When a third party opens a feature-request / integration issue on this repo that overlaps existing functionality, the user's preferred response pattern is: (1) state candid capacity status, (2) describe what this repo already does with pointers to exact module paths, (3) call out *why* it's genuinely different (architecture, not just implementation), (4) list 2-3 concrete integration seams without committing, (5) ask the requester to post a written integration plan for async review. Do NOT be dismissive — the user explicitly overrode a colder draft with "I don't want to be too cold... no harm in supporting him."
- **[pattern]** Proxy smoke-test after restart: `curl -s -o /dev/null -w "status=%{http_code} proto=%{http_version} time=%{time_total}s\n" http://<bind>:<port>/v1/messages -X POST -H "x-api-key: bogus" …` — a `401` response with `proto=1.0` (curl's client side is HTTP/1.0, the proxy upstream is still HTTP/2) confirms end-to-end routing. To verify the **upstream** http_version, call `claude_hooks.proxy.forwarder.forward()` in-process and read `result.stats['http_version']` — should be `HTTP/2` after the httpx migration.
- **[gotcha]** Windows `C:\Windows\system32\timeout.exe` is a `sleep`/countdown utility — NOT a GNU-coreutils `timeout` command wrapper. A pre-commit hook that uses `timeout 30 caliber …` will find the wrong binary first on PATH and hang. In portable sh scripts, explicitly probe for `/usr/bin/timeout` (Git-Bash coreutils) or `gtimeout`, and fall back to relying on `CALIBER_CLAUDE_CLI_TIMEOUT_MS` when no GNU `timeout` exists.
- **[gotcha]** Caliber's stock pre-commit hook is blocking and calls BOTH `caliber refresh` (~70s LLM call) AND `caliber learn finalize` (non-auto; can run 10+ min on sessions with ≥240 events) — a 20+ min commit stall is the default on active repos. Make the hook fire-and-forget: run `caliber refresh --quiet` backgrounded with `</dev/null >/dev/null 2>&1 &`, drop `caliber learn finalize` from the commit path entirely (SessionEnd already runs `--auto`), and bound with `timeout 30` + `CALIBER_CLAUDE_CLI_TIMEOUT_MS=60000`. Refreshed docs land in the *next* commit, which is acceptable.
- **[gotcha]** Running `ssh pandorum 'time git commit ...'` invokes Windows `cmd.exe`'s `time` builtin, which prompts for a new time and returns `The system cannot accept the time entered.` — not the elapsed-time wall clock. For timing on pandorum, write a small `.sh` benchmark script (capturing `date +%s%N` before/after), `scp` it, and run via `ssh pandorum '"C:/Program Files/Git/bin/sh.exe" C:/Users/manni/hook-bench.sh'`. Inline `bash -c "…\$(…)…"` with nested `$()` via ssh hits quote-escaping bugs — prefer a file.
- **[gotcha]** `ssh pandorum 'bash -c "cd /c/Users/…"'` fails with `No such file or directory` — the default ssh shell is an MSYS bash whose `/c/...` translation is not what you expect. Use the Git-Bash binary explicitly: `ssh pandorum '"C:/Program Files/Git/bin/sh.exe" -c "cd /c/Users/manni/…"'`, or write the commands to a file and run the file.
- **[pattern]** To deploy a changed git hook across both solidPC and pandorum (3 repos), write ONE portable `.sh` once to `/tmp/`, back up each existing hook with a timestamped `.bak-YYYYMMDD-HHMMSS` suffix (`cp .git/hooks/pre-commit .git/hooks/pre-commit.bak-$(date +%Y%m%d-%H%M%S)`), then `scp` to each remote location. `.git/hooks/` is NOT tracked in git so deployment is per-clone.
- **[gotcha]** `git reset --soft HEAD~1` after timing-test commits can accidentally walk back past *real* commits if you run it more than once — always check `git reflog` and `git status --short | wc -l` before deciding the tree is clean. Restore with `git reset --soft origin/<branch>` to snap HEAD back to the pushed tip without losing working-tree content.

- **[correction]** User does not have and will not use an Anthropic API key — attempts to point Caliber at `OPENAI_API_KEY` + a local Ollama `OPENAI_BASE_URL` are only valid if `CALIBER_MODEL` is set (without it, Caliber picks its default `gpt-5.4-mini` and 404s on Ollama). `CALIBER_FAST_MODEL` MUST be set separately — validateModel probes the fast-model too, and defaults to `gpt-5.4-mini` for `openai` provider regardless of `CALIBER_MODEL`. Same applies to any OpenAI-compatible local endpoint.
- **[gotcha]** Ollama's OpenAI-compatible endpoint (`/v1/chat/completions`) silently clamps context to the model's default `num_ctx=4096` unless you either (a) pass `options: {"num_ctx": N}` in the request body — NON-standard OpenAI field, accepted by Ollama — or (b) build a derived model with `PARAMETER num_ctx N` in a Modelfile and `ollama create <name>-<ctx>k -f …`. Caliber's OpenAI client does NOT set `options.num_ctx`, so when pointing Caliber at Ollama you MUST create a dedicated `<model>-128k` tag or refresh will fail with `No JSON found in LLM response` — the model was truncated mid-JSON at 4k tokens even though `max_completion_tokens=10240`.
- **[pattern]** To run Caliber against a local Ollama endpoint for comparison/dev: `mv ~/.caliber/config.json ~/.caliber/config.json.bak` → `export OPENAI_API_KEY=ollama; export OPENAI_BASE_URL=http://<host>:11433/v1; export CALIBER_MODEL=<derived-model-with-large-ctx>; export CALIBER_FAST_MODEL=<same>` → run, then `mv` the backup back. The config file takes precedence over env vars (see `loadConfig` → `readConfigFile` first, `resolveFromEnv` second), so the file MUST be moved aside, not just overridden with env.
- **[pattern]** Before running `caliber init` / `regenerate` / `refresh` (even with `--dry-run`) back up the four files it is known to mutate: `.claude/settings.json`, `.claude/settings.local.json`, `CLAUDE.md`, `AGENTS.md` → `*.init-test-bak`. `--dry-run` does NOT guarantee zero writes on `init`; clear any stale `.caliber/learning/finalize.lock` first or the run exits early.
- **[gotcha]** `caliber regenerate` refuses with `No existing config found. Run /usr/local/bin/caliber init first.` when `.caliber/manifest.json` is missing — `regenerate` is an update-over-existing, not a first-run command. Running it on a repo that only has Caliber pre-commit hooks but no `init` artifacts will always fail.
- **[gotcha]** When A/B-testing Caliber against different local Ollama models, the first hit is **cold-load VRAM** and adds minutes of latency. Warm the model with a tiny `curl /v1/chat/completions` (1 token) before timing the real run, otherwise the first model in the sequence looks 2-3x slower than the rest.
- **[pattern]** Caliber's project context comes from `collectFingerprint()` in `/usr/local/lib/node_modules/@rely-ai/caliber/dist/bin.js` — it reads only `README.md`, `AGENTS.md`, `CLAUDE.md`, `.claude/settings.json`, a git-tracked file *tree* (no file contents beyond the anchors), and LLM-derived language/framework tags. Local models without native grounding tools therefore see only names + the four anchor files, which is why they hallucinate skill file refs. To fix quality, front Ollama with a grounding proxy that exposes `read_file`/`grep`/`glob`/`list_files` as OpenAI tool-calls, pre-stuffs anchor files up to ~50KB, and runs an agent loop (~35 iterations cap).
- **[gotcha]** Gemma4-98e (via Ollama `/v1/chat/completions`) DOES honor OpenAI-style `tools`/function-calling — `finish_reason: "tool_calls"` comes back with a valid `tool_calls[].function.arguments` JSON string. Probe with a 1-shot `{"tools":[...],"max_completion_tokens":200}` request before assuming a local model needs a prompt-only workaround. Qwen3:14b in contrast silently produces narrative prose instead of structured JSON output on caliber init, even with large `num_ctx` — resulting in `Model responded but output was not valid JSON`.
- **[gotcha]** Ollama `ollama create` rejects Modelfile templates with undefined Go-template functions (`sub`, `add`) — error: `template error: template: :40: function "sub" not defined`. Available built-ins are limited to `slice`, `len`, `eq`, `ne`, `gt`, `range`, `if`. Use `{{- $last := eq (len (slice $.Messages $i)) 1 -}}` to detect the last message instead of `(sub (len .Messages) 1)`. Also: an `{{- end }}` without a matching opener fails with `unexpected {{end}}` and aborts the build — template tag counts must balance exactly.
- **[gotcha]** When running an agent-loop proxy that forwards `tool_calls` back to the model, the assistant's echoed `content` field — which models sometimes emit as template fragments / partial JSON alongside the structured `tool_calls[]` — will derail the next iteration's generation (model parrots the garbage and produces a 3-char response like `'\n```'`). Fix: strip `content` to `None` on any assistant message that carries `tool_calls` before re-sending (`clean_msg = dict(msg); clean_msg['content'] = None`). Dump outgoing payloads with a `CALIBER_GROUNDING_DUMP_DIR` env var to confirm.
- **[pattern]** For long-running background processes (e.g. local proxies) started during a Claude Code session, use the Bash tool's `run_in_background: true` facility rather than shell `nohup ... &` + `disown` — plain shell backgrounding gets reaped when the Bash tool's wrapper shell exits (`httpcore.connection: close.complete` immediately after command finishes), whereas the harness's backgroundTaskId survives across subsequent tool calls. Health-check with `curl -sf http://127.0.0.1:<port>/health` a few seconds after start.
- **[gotcha]** When reloading nginx inside a docker compose project, `cd /shared/config/<proj> && docker compose exec -T <svc> nginx -s reload` changes the session's `cwd` — the Bash tool emits `Shell cwd was reset to <orig>` on stderr. That's cosmetic, not a failure; use exit 0 + `nginx -t` output to confirm reload succeeded.
