# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** `CLAUDE_CODE_SIMPLE=1` env var breaks OAuth authentication in `claude -p` — causes "Not logged in" even when `claude auth status` reports logged in. Caliber's `spawnClaude` sets this var, which is the root cause of `caliber learn finalize` failures with claude-cli provider. Fix: patch `/usr/local/lib/node_modules/@rely-ai/caliber/dist/bin.js` to `delete env.CLAUDE_CODE_SIMPLE` instead of setting it to "1". This patch must be re-applied after every `npm install -g @rely-ai/caliber` upgrade.
- **[gotcha]** Caliber session state (`/.caliber/learning/state.json` and `current-session.jsonl`) can accumulate stale events across sessions. If `eventCount` grows large without `lastAnalysisTimestamp` advancing, reset both files manually: write `{"sessionId":null,"eventCount":0,"lastAnalysisTimestamp":null,"lastAnalysisEventCount":0}` to state.json and truncate the jsonl.
- **[env]** Caliber error diagnostics live in multiple files: `.caliber/learning/last-error.json` (learn finalize errors), `.caliber/last-refresh-error.json` (refresh errors), `.caliber/error-log.md` (generation errors with raw LLM output). Check all three when debugging.
- **[gotcha]** `CLAUDE_CODE_SIMPLE=1` env var breaks OAuth authentication when running `claude -p`. Claude CLI returns "Not logged in · Please run /login" even though `claude auth status` reports `loggedIn: true`. Any tool that spawns `claude -p` (e.g. Caliber) must NOT set this env var. Workaround: `delete env.CLAUDE_CODE_SIMPLE` before spawning, or use `env -u CLAUDE_CODE_SIMPLE claude -p`.
- **[gotcha]** Caliber's `finalize.lock` at `.caliber/learning/finalize.lock` can get stuck after a failed finalize run. If `caliber learn finalize` reports "another finalize is in progress" but nothing is actually running, manually delete the lock file.
- **[pattern]** When Caliber's `claude-cli` provider fails with auth errors, the root cause is usually an env var conflict — not missing credentials. Check the spawned process environment before assuming the OAuth token is invalid.
- **[env]** Caliber stores its LLM provider config at `~/.caliber/config.json` (global) and `.caliber/config.json` (project). The `provider: "claude-cli"` setting means it shells out to `claude -p` — no API key needed, but the CLI must be logged in and the spawn environment must be clean.
- **[gotcha]** A running Claude Code session holds the Caliber binary in memory — patching `/usr/local/lib/node_modules/@rely-ai/caliber/dist/bin.js` does NOT affect sessions that were started before the patch. Resumed sessions will still hit the old auth bug. Only newly-spawned sessions (or explicit re-invocation of `caliber` from the CLI) pick up the patch.
- **[gotcha]** `claudemem` binary at `/usr/local/bin/claudemem` is actually a symlink to the `mnemex` npm package (`../lib/node_modules/mnemex/dist/index.js`). Skills complaining "claudemem not installed" usually mean the index is missing or outdated, NOT the binary. Verify with `claudemem status` — look for `exists=false` or `Index outdated (v1 -> v2)`.
- **[fix]** When a code-analysis skill reports claudemem unavailable, run `claudemem status` in the project root. If it shows `Index outdated (v1 -> v2). Missing features`, run `claudemem index -y --force` to upgrade the index. Indexing large projects takes minutes (anycubic ~90min, lightseek ~40min) — run in background with `run_in_background: true`.
- **[env]** `claudemem` indexes are per-project and must be built separately for every working directory. The list of OpenWolf-registered projects is at `/root/.openwolf/registry.json` — iterate its `projects[].root` to bulk-index all known project roots.
- **[pattern]** To locate a Caliber (or similar npm-installed tool's) upstream repo, read `repository.url` from `$(npm root -g)/<pkg>/package.json` via `node -e`. For `@rely-ai/caliber` the actual repo is `caliber-ai-org/ai-setup` on GitHub — not discoverable from the package name.
- **[convention]** Project-specific patches to third-party npm globals live under `patches/` with (a) a markdown doc explaining root cause + upstream issue links + removal criteria, and (b) an idempotent `apply-*.sh` shell script. The script must detect already-patched state and exit cleanly when pattern is not found.
- **[correction]** User does not have and will not use an Anthropic API key — Caliber must use the `claude-cli` provider (seat-based billing via Claude Code subscription), never the API provider.
- **[env]** `claudemem` (mnemex) reads its config from `~/.claudemem/config.json` and uses `embeddingProvider: "ollama"` with `ollamaEndpoint: "http://192.168.178.2:11433"` and model `snowflake-arctic-embed2`. Pandorum uses the same config (via `C:\Users\manni\.claudemem\config.json`). Verify the model is resident with `curl -s http://192.168.178.2:11433/api/ps`.
- **[gotcha]** Two unrelated CLIs both ship a `rtk` binary: [rtk-ai/rtk](https://github.com/rtk-ai/rtk) (the one used by Claude Code hooks for command rewriting; provides `rtk rewrite`) and "Rust Type Kit" (cargo-installed at `~/.cargo/bin/rtk`, version 0.1.0, has no `rewrite` subcommand). If `rtk --version` prints `0.1.0` you have the wrong one — remove it and install rtk-ai via `curl -fsSL https://raw.githubusercontent.com/rtk-ai/rtk/refs/heads/master/install.sh | sh` (Linux) or the Windows zip from GitHub releases. The `rtk rewrite` subcommand requires rtk-ai version >= 0.23.0.
- **[pattern]** PowerShell quoting through `ssh pandorum "powershell -Command ..."` is fragile (quadruple-escaping rapidly breaks down). Instead: write the script to a local `.ps1` file, `scp` it to the Windows host, then `ssh pandorum "powershell -NoProfile -ExecutionPolicy Bypass -File <path>"` and delete it after. Reliable and debuggable.
- **[convention]** External-binary hooks (e.g. `rtk_rewrite`) must be opt-in (`enabled: false` by default) AND must no-op silently when the binary is missing or the wrong version. Core hooks (recall/store/session/safety) stay stdlib-only. This split lets the same config work on hosts with and without the optional binary.
- **[gotcha]** Claude Code does NOT re-fire `PreToolUse` after a hook returns `updatedInput` — the rewritten command runs immediately without going through the hook chain a second time. Multi-stage hooks that need to inspect the *rewritten* form (e.g. rtk rewrite + safety scan) must compose all stages in a single response; do not assume a re-fire will give later stages another pass.
- **[pattern]** To live-test a hook handler without cycling through Claude Code, pipe a JSON event to the entry-point shim: `echo '{"hook_event_name":"PreToolUse","tool_name":"Bash","tool_input":{"command":"..."},"cwd":"..."}' | bin/claude-hook PreToolUse`. The shim's stdout is the exact response Claude Code will see.
- **[correction]** "delete it" in this project means delete the *specific artefact just discussed* (e.g. the local backup file), NOT the whole feature/module that was the broader topic. When an instruction is ambiguous between a narrow and broad target, ask before deleting source files / tests / config blocks. The user had to revert a deletion of `stop_guard.py` because the agent assumed "delete it" meant the feature, when it referred to the local backup file from the prior turn.
- **[gotcha]** Windows git on pandorum reports spurious file-mode changes (100755 ↔ 100644) on tracked files. Disable with `git config core.filemode false` in the repo. Without this, `git pull` on pandorum will complain about local mods that aren't real content changes.
- **[fix]** If `git pull` on pandorum aborts with "untracked working tree files would be overwritten", check whether the untracked file is byte-identical to the upstream version (same script may have been generated locally and then committed upstream). Compare with CRLF-normalised content (PowerShell: `(Get-Content -Raw) -replace [char]13+[char]10, [char]10`) before assuming a real conflict.
- **[gotcha]** Heredoc-piped PowerShell scripts containing `\`r\`n` backtick-escapes get the backticks stripped by bash before reaching PowerShell. Use `[char]13` and `[char]10` (or write the .ps1 with non-bash tooling) when you need explicit CR/LF in a script delivered via heredoc.
- **[convention]** When extending the `stop_guard` (or any pattern-matching hook) with new behaviour, also test the *meta-context escape*: a message that quotes or documents a trigger phrase (`"pre-existing issue"`) must NOT fire the hook. The `MetaContextEscapeTests` class in `tests/test_stop_guard.py` is the reference pattern — reuse it for similar hooks.
- **[gotcha]** `claudemem <subcmd> --help` and `claudemem --agent <subcmd> --help` both just re-print the global help — they do NOT show subcommand-specific usage. Use `claudemem <subcmd> help` (no dashes) instead, e.g. `claudemem hooks help` prints the subcommands for `hooks`. `claudemem --agent watch` additionally spawns a long-running daemon; if run unintentionally, kill with `pkill -f 'claudemem.*watch'`.
- **[pattern]** `claudemem hooks install` sets up a post-commit git hook at `.git/hooks/post-commit` that runs `claudemem index --quiet` in the background after every commit. It's per-project — install it once per repo. Verify with `claudemem hooks status`. For claude-hooks, this covers committed changes; mid-session uncommitted edits are handled by the `claudemem_reindex` hook in this project.
- **[pattern]** User's terse prompts ("delete it", "revert it", single-word answers) are frequently symptoms of Claude Code TUI input lag — keyboard echo is slowed and chars get dropped, so the user truncates mid-sentence. Default to asking "which one?" when pronouns point to multiple candidates; do NOT optimise for fewer round-trips at the cost of guessing wrong. Per-turn hook latency also compounds the client lag, so be conservative when adding new hook work.
